# 术后随访 AI 协同系统 - 需求文档

## 1. 项目概述

### 1.1 项目名称
术后随访 AI 协同系统（Post-Surgery Follow-up AI System）

### 1.2 项目目标
服务患者术后随访场景，通过 AI 健康顾问与患者对话收集症状信息，生成结构化病历供医生参考，实现高效的术后随访管理。

### 1.3 核心价值
- **患者端**：便捷的术后健康咨询，AI 实时健康建议，复查预约
- **医生端**：接收 AI 结构化病历，高效诊疗决策，批量管理患者和复查预约

### 1.4 系统定位
基于 **RAG + 预训练 LLM** 的医患协同系统
- 使用预训练 GLM-4.7-Flash（未微调）
- 通过 RAG 检索医学知识库补充 AI 回答
- AI 无法回答时，调用 LLM 自身知识生成合理建议

## 3. 功能归属划分

### 3.1 总体划分

| 归属 | 负责范围 | 技术栈 |
|------|----------|--------|
| **前端** | 患者端 UI、医生端 UI | React/Vue |
| **AI 服务** | 对话引擎、混合检索、病历生成 | Python/gRPC |
| **后端** | 业务逻辑、数据库、API 网关 | Python/FastAPI |

---

### 3.2 前端负责（患者端）

| 功能 | 前端实现 | 调用关系 |
|------|----------|----------|
| 发起随访 | 按钮点击 → 调用后端 API | → 后端 |
| 流式对话 | SSE 接收 → 实时显示 AI 回复 | → AI 服务 |
| 结束对话 | 按钮点击 → 调用后端 API | → 后端 |
| 查看 AI 建议 | 界面展示 | - |
| 发起复查预约 | 表单提交 → 后端 API | → 后端 |

**前端实现要点**：
- 流式响应使用 **SSE (Server-Sent Events)**
- 实时显示 AI 正在输入的效果
- 处理网络中断重连

---

### 3.3 前端负责（医生端）

| 功能 | 前端实现 | 调用关系 |
|------|----------|----------|
| 添加患者 | 表单提交 | → 后端 API |
| 查看病历列表 | 列表展示 | → 后端 API |
| 查看病历详情 | 结构化展示 | → 后端 API |
| 查看对话记录 | 对话气泡展示 | → 后端 API |
| 填写医疗建议 | 表单提交 | → 后端 API |
| 发出复查预约 | 表单提交 | → 后端 API |
| 管理复查预约 | CRUD 列表 | → 后端 API |

---

### 3.4 AI 服务负责

| 功能 | 说明 | 接口 |
|------|------|------|
| 会话管理 | 创建/结束/查询会话 | CreateSession, EndSession, GetSessionHistory |
| 流式对话 | 接收消息，返回流式响应 | Chat (双向流) |
| 手术类型确认 | 首次对话询问手术类型 | 内部逻辑 |
| 症状收集 | 主动询问症状细节 | 提示词 |
| 混合检索 | 本地 RAG + 智谱知识库 + LLM | 检索 + 生成 |
| 病历生成 | 输出结构化病历 | EndSession 返回 |

**AI 服务接口**：
```protobuf
rpc CreateSession(CreateSessionRequest) returns (CreateSessionResponse);
rpc Chat(stream ChatRequest) returns (stream ChatResponse);  // 流式
rpc EndSession(EndSessionRequest) returns (EndSessionResponse);
rpc GetSessionHistory(GetSessionHistoryRequest) returns (GetSessionHistoryResponse);
rpc CreateFollowupAppointment(CreateFollowupRequest) returns (CreateFollowupResponse);
```

---

### 3.5 后端负责

| 功能 | 说明 | 数据库 |
|------|------|--------|
| 患者管理 | CRUD | patients 表 |
| 对话历史存储 | 调用 AI 服务获取后存储 | chat_history 表 |
| 结构化病历存储 | 调用 AI 服务获取后存储 | medical_reports 表 |
| 复查预约管理 | CRUD + 状态流转 | appointments 表 |
| 医生建议存储 | 医生填写的建议 | medical_reports 表 |
| 消息推送 | WebSocket/短信/邮件 | - |

**后端调用 AI 服务流程**：
```
1. 前端 → 后端：发起随访
   后端 → AI 服务：CreateSession

2. 前端 → 后端：发送消息
   后端 → AI 服务：Chat (流式)
   AI 服务 → 后端：流式响应
   后端 → 前端：SSE 转发

3. 前端 → 后端：结束对话
   后端 → AI 服务：EndSession
   AI 服务 → 后端：结构化病历
   后端 → 数据库：存储病历

4. 前端 → 后端：创建预约
   后端 → AI 服务：CreateFollowupAppointment
   后端 → 数据库：存储预约
```

## 5. AI 对话策略

### 5.1 AI 角色

| 阶段 | 角色 | 职责 |
|------|------|------|
| 对话阶段 | 健康顾问 | 确认手术、询问症状、提供 RAG 健康建议 |
| 报告阶段 | 专科医生 | 生成结构化病历、给出辅助判断 |

### 5.2 对话流程

```
┌─────────────────────────────────────────────────────────────┐
│                      AI 对话流程                            │
├─────────────────────────────────────────────────────────────┤
│                                                             │
│  【阶段1：确认手术类型】  ← 新增                            │
│   AI 询问患者所做的手术类型                                  │
│   示例：「您好！我是您的术后随访助手。请问您做的是什么手术？」│
│                                                             │
│           ▼                                                  │
│  【阶段2：开场】                                            │
│   AI 根据手术类型定向检索，准备个性化问题                    │
│   示例：「好的，您进行的是阑尾切除手术。请描述一下您        │
│         术后有什么不适吗？」                                 │
│                                                             │
│           ▼                                                  │
│  【阶段3：症状收集】                                        │
│   AI 询问症状细节、持续时间、严重程度等                     │
│   - 定向 RAG 检索提供健康建议                               │
│   - LLM 知识补充（文档不足时）                             │
│   - 适时询问「还有其他症状吗？」                            │
│                                                             │
│           ▼                                                  │
│  【阶段4：确认结束】                                        │
│   AI 询问是否还有补充                                        │
│   示例：「您还有其他症状需要补充吗？」                        │
│                                                             │
│           ▼                                                  │
│  【阶段5：生成报告】                                        │
│   AI 整合对话内容，生成结构化病历                          │
│   切换为「专科医生」角色，给出辅助判断                      │
```

### 5.4 AI 回复策略

| 策略 | 描述 |
|------|------|
| 定向检索 | 根据手术类型优先检索相关文档 |
| RAG + LLM | 有文档用文档，文档不足 LLM 补充 |
| 引导询问 | 主动询问症状细节（时间、位置、程度等） |
| 症状确认 | 适时确认患者描述的症状 |
| 结束询问 | 在信息足够时询问是否还有补充 |
| 主动提示 | 提醒患者何时需要就医 |

## 7. RPC 接口设计

### 7.1 CreateSession - 创建会话
```protobuf
rpc CreateSession(CreateSessionRequest) returns (CreateSessionResponse);

message CreateSessionRequest {
  string patient_id = 1;        // 患者ID
  string patient_name = 2;       // 患者姓名
  string surgery_date = 3;      // 手术日期
}

message CreateSessionResponse {
  string session_id = 1;        // 会话ID
  string created_at = 2;        // 创建时间
  string welcome_message = 3;   // 欢迎语（询问手术类型）
}
```

### 7.2 Chat - 流式对话
```protobuf
rpc Chat(stream ChatRequest) returns (stream ChatResponse);

message ChatRequest {
  string session_id = 1;       // 会话ID
  string message = 2;          // 患者消息
  bool is_end = 3;             // 是否结束对话
}

message ChatResponse {
  string content = 1;          // AI 回复内容（流式）
  bool is_final = 2;           // 是否为最终回复
  string reference = 3;        // RAG 检索参考（可选）
}
```

### 7.3 EndSession - 结束会话
```protobuf
rpc EndSession(EndSessionRequest) returns (EndSessionResponse);

message EndSessionRequest {
  string session_id = 1;        // 会话ID
}

message EndSessionResponse {
  MedicalReport report = 1;     // 结构化病历
  string status = 2;            // 状态
}

message MedicalReport {
  PatientInfo patient = 1;           // 患者信息
  string chief_complaint = 2;         // 主诉
  string present_illness = 3;         // 现病史
  PostSurgeryStatus surgery_status = 4; // 术后恢复情况
  PastHistory past_history = 5;       // 既往史
  AIAnalysis ai_analysis = 6;         // AI 辅助分析
  DoctorAdvice doctor_advice = 7;     // 医生处理意见（可为空）
  SessionInfo session_info = 8;       // 会话信息
}

message PatientInfo {
  string name = 1;
  string gender = 2;
  int32 age = 3;
  string surgery_type = 4;
  string surgery_date = 5;
  string doctor_name = 6;
}

message PostSurgeryStatus {
  string wound_healing = 1;    // 良好/红肿/渗液/裂开
  string temperature = 2;       // 正常/低热/发热
  string appetite = 3;         // 正常/减退/恶心呕吐
  string activity = 4;         // 正常/乏力/受限
}

message PastHistory {
  string allergy = 1;
  string chronic_disease = 2;
  string medication = 3;
}

message AIAnalysis {
  string health_advice = 1;
  bool alert_flag = 2;         // 异常预警
  string alert_reason = 3;
}

message DoctorAdvice {
  string medical_advice = 1;
  string followup_items = 2;
  string next_visit_date = 3;
  bool need_followup = 4;
}

message SessionInfo {
  int32 conversation_rounds = 1;
  string submitted_at = 2;
}
```

### 7.4 GetSessionHistory - 获取对话历史

```protobuf
rpc GetSessionHistory(GetSessionHistoryRequest) returns (GetSessionHistoryResponse);

message GetSessionHistoryRequest {

message GetSessionHistoryResponse {
```

## 8. 后端对接说明

### 8.1 前后端职责划分

| 归属 | 负责范围 |
|------|----------|
| **前端** | 患者端 UI、医生端 UI |
| **AI 服务** | 对话引擎、混合检索、病历生成 |
| **后端** | 业务逻辑、数据库、API 网关 |

### 8.2 前端功能（患者端）

| 功能 | 前端实现要点 | AI 服务接口 |
|------|-------------|-------------|
| 发起随访 | 调用 CreateSession，显示欢迎语 | CreateSession |
| 流式对话 | 接收 Chat 流式响应，边生成边显示 | Chat (流式) |
| 结束对话 | 调用 EndSession | EndSession |
| 查看 AI 建议 | 显示 AI 回复内容 | - |
| 发起复查预约 | 调用后端 API | CreateFollowupAppointment |

**前端实现要点**：
- 流式响应需要使用 **Server-Sent Events (SSE)** 或 **WebSocket**
- 界面需要支持实时显示 AI 正在输入的效果
- 需要处理网络中断重连

### 8.3 前端功能（医生端）

| 功能 | 前端实现要点 | 后端接口 |
|------|-------------|---------|
| 添加患者 | 表单提交 | 后端 API |
| 查看病历列表 | 列表展示 | 后端 API |
| 查看病历详情 | 结构化展示 | 后端 API |
| 查看对话记录 | 对话气泡展示 | GetSessionHistory |
| 填写医疗建议 | 表单提交 | 后端 API |
| 发出复查预约 | 表单提交 | CreateFollowupAppointment |
| 管理复查预约 | 列表、修改、取消 | 后端 API |

### 8.4 AI 服务功能

| 功能 | 说明 | 内部实现 |
|------|------|----------|
| 会话管理 | CreateSession, EndSession, GetSessionHistory | 会话状态存储 |
| 流式对话 | Chat 接口，流式返回 | LLM 调用、混合检索 |
| 手术类型确认 | 第一步询问手术类型 | 提示词引导 |
| 症状收集 | 主动询问症状细节 | 提示词 + RAG |
| 混合检索 | 本地 RAG + 智谱知识库 + LLM | 检索 + 生成 |
| 病历生成 | 结构化病历输出 | 提示词模板 |

**AI 服务接口设计考虑**：
- 流式输出支持前端实时显示
- 返回内容包含状态标记（is_end, is_final 等）
- 方便后端判断对话阶段

### 8.5 后端功能

| 功能 | 说明 | 关联接口 |
|------|------|----------|
| 患者管理 | 增删改查 | 前端表单 |
| 对话历史存储 | 存储患者与 AI 的对话 | GetSessionHistory |
| 结构化病历存储 | 存储 AI 生成的病历 | EndSession |
| 复查预约管理 | 业务逻辑 + 数据库 | CreateFollowupAppointment |
| 医生建议存储 | 存储医生的医疗建议 | 前端表单 |
| 消息推送 | 向患者推送医生建议/预约提醒 | WebSocket/短信等 |

**后端对接要点**：
- AI 服务返回的数据需要立即存储（对话历史、病历）
- 复查预约需要事务处理
- 需要处理 AI 服务的超时和错误

### 8.6 AI 服务提供的接口

| 接口 | 用途 | 后端调用时机 |
|------|------|-------------|
| CreateSession | 创建会话 | 患者发起随访时 |
| Chat | 流式对话 | 对话过程中多次调用 |
| GetSessionHistory | 获取对话历史 | 对话结束后，获取完整对话 |
| EndSession | 结束会话，获取病历 | 患者确认结束时 |
| CreateFollowupAppointment | 创建复查预约 | 需要预约时 |

### 8.7 推荐后端调用流程

```
1. 患者发起随访
   后端调用 CreateSession → 返回 session_id

2. 患者与 AI 对话
   后端调用 Chat (流式) → 患者发送消息 → AI 返回流式响应

3. 患者确认结束
   后端调用 EndSession → 返回结构化病历
   后端存储对话历史 + 病历到数据库

4. 需要复查预约
   后端调用 CreateFollowupAppointment
```

### 8.8 会话状态说明

| 状态 | 含义 | 后端注意事项 |
|------|------|-------------|
| WAITING_SURGERY | 等待确认手术类型 | 对话刚开始 |
| ACTIVE | 对话进行中 | 可多次调用 Chat |
| COMPLETED | 对话已结束 | 调用 EndSession 后 |
| CANCELLED | 对话已取消 | 异常情况 |

### 8.9 会话存储

| 方案 | 说明 | 适用场景 |
|------|------|----------|
| 内存存储 | 简单快速 | 单实例部署 |
| Redis 存储 | 支持分布式 | 多实例部署 |
| 文件存储 | 持久化 | 开发/测试环境 |

**建议**：生产环境使用 Redis，支持分布式部署和会话共享

### 8.10 流式协议

| 协议 | 说明 | 前端实现 |
|------|------|----------|
| SSE (Server-Sent Events) | 单向流，推荐 | EventSource API |
| WebSocket | 双向通信 | WebSocket API |

**建议**：Chat 流式对话使用 **SSE**，更简单且符合单向数据流场景

### 8.11 错误处理

| 错误码 | 说明 | 处理建议 |
|--------|------|----------|
| SESSION_NOT_FOUND | 会话不存在 | 重新创建会话 |
| SESSION_EXPIRED | 会话已超时 | 提示用户重新开始 |
| INVALID_REQUEST | 请求参数错误 | 检查参数 |
| AI_SERVICE_ERROR | AI 服务内部错误 | 重试或提示 |
| RAG检索失败 | 检索异常 | 回退到 LLM 自身知识 |

### 8.12 认证机制

| 方案 | 说明 |
|------|------|
| API Key | 简单场景 |
| JWT Token | 推荐，支持过期和验证 |
| mTLS | 高安全要求 |

**建议**：后端与 AI 服务通信使用 **JWT Token**，确保安全性

### 8.13 注意事项

1. **会话超时**：AI 服务可能设置会话超时（如 30 分钟无活动自动结束）
2. **并发控制**：同一 session_id 不建议并发调用 Chat
3. **错误处理**：后端需要处理 AI 服务返回的错误码
4. **重试机制**：网络异常时建议重试，但需保证幂等性

```python
def hybrid_retrieve(query, surgery_type):
    # 1. 本地 RAG 检索（术后护理文档）
```

## 12. 附录

### A. 对话示例

以下是对话示例，展示 AI 如何结合 RAG 检索和 LLM 自身知识进行回答：

```
患者：伤口疼痛

AI（RAG 检索到相关文档 + LLM 优化）：
「深呼吸和翻身时疼痛加重，说明伤口周围的肌肉和皮肤组织受到了牵拉，
这通常是因为切口愈合过程中的正常反应。

请问伤口周围有没有出现红肿、发热的情况？或者按压伤口时感觉更痛？」

患者：没有

AI：
「伤口周围没有红肿和发热，说明没有明显的感染迹象，这让我们放心了不少。
目前你的排便情况如何？有没有排气（放屁）或者排便？」

患者：没有

AI（LLM 自身知识）：
「这种情况我们需要警惕一下。阑尾手术后如果超过24小时仍未排气，
且伴有持续加重的伤口疼痛，可能是肠粘连的早期迹象，或者是肠道蠕动受抑制。

请问除了伤口痛，你有没有感觉到肚子胀气（肚子鼓鼓的）？」

患者：没有

AI：
「肚子没有胀气是好消息，说明肠道蠕动可能已经开始恢复，
不需要太担心肠梗阻的问题。

既然疼痛持续且在深呼吸、翻身时加重，除了按时服用止痛药外，
你现在躺着的姿势是什么（比如平躺、侧卧）？」

...

（AI 持续收集症状，结合文档和 LLM 知识给出个性化建议）
```

**示例特点**：
- 检索到文档时：基于文档内容 + LLM 补充细节
- 未检索到文档时：直接调用 LLM 自身医学知识
- 主动询问症状细节
- 适时确认患者描述
- 提供具体的缓解建议
