## 一、分工

## 二、开发者 A

### GitHub上要做的事

1. **邀请开发者B和C**
   - 打开 https://github.com/Felicity-byte/-/settings/access
   - 点击 **Add people**
   - 输入开发者B/C的GitHub用户名或邮箱
   - 选择 **Write** 权限，发送邀请

2. **审核开发者B的PR**
   - 收到PR通知后，打开仓库
   - 点击 **Pull requests** → 查看PR
   - 点击 **Files changed** 查看代码改动
   - 确认无误后点击 **Merge pull request**

### 日常开发

```bash
# 拉取最新代码
git pull origin main

# 修改代码...
# ...

# 添加所有修改的文件
git add .

# 提交并写上描述
git commit -m "feat: 描述"

# 推送到远程main分支
git push origin main
```

## 三、开发者 B

### GitHub上要做的事

1. **接受邀请**
   - 查收邮件，点击邀请链接
   - 或访问 https://github.com/Felicity-byte/- 接受邀请

2. **创建Pull Request**
   - 推送代码后，打开GitHub仓库
   - 点击 **Compare & pull request**
   - 填写描述，点击 **Create pull request**
   - 等待开发者A审核

### 第一次使用

详见 [开发者B环境搭建指南.md](./开发者B环境搭建指南.md)

### 开发新功能

```bash
# 1. 切换到main分支
git checkout main

# 2. 拉取最新代码
git pull origin main

# 3. 创建新分支（每个功能一个分支）
git checkout -b feature/功能名

# 4. 开发（只修改Frontend目录下的文件）
# ...

# 5. 添加所有修改的文件
git add .

# 6. 提交并写上描述
git commit -m "feat: 描述"

# 7. 推送到远程分支
git push -u origin feature/功能名

# 8. 在GitHub创建Pull Request，等待开发者A审核
```

### 功能没开发完

```bash
# 继续在当前分支开发，不需要创建新分支

# 添加所有修改的文件
git add .

# 提交
git commit -m "feat: 继续开发功能"

# 推送到远程分支
git push

# 继续开发...
# 直到功能完成，创建PR
```

### PR被合并后

```bash
# 切换回主分支
git checkout main

# 拉取最新代码（包含刚合并的内容）
git pull origin main

# 删除本地已合并的功能分支
git branch -d feature/功能名
```

### 分支规则

| 情况 | 操作 |
| --- | --- |
| 开发新功能 | 创建新分支 |
| 同一功能多次提交 | 用同一个分支 |
| 功能没开发完 | 继续在原分支开发 |
| PR被合并后 | 删除本地分支 |

## 四、开发者 C

### 职责

- 负责整个项目的测试工作
- 不需要写代码，只需要运行测试
- 发现问题后反馈给开发者A

### 第一次使用

详见 [主要测试流程.md](../test/Gtests/主要测试流程.md)

### 测试流程

```bash
# 1. 拉取最新代码
git fetch origin
git checkout main
git reset --hard origin/main

# 2. 启动AI服务（新终端）
cd D:\Trea\medical systems\CareLinkAgent\GlmAI
python connect/server.py

# 3. 运行测试（另一个终端）
cd D:\Trea\medical systems\CareLinkAgent\test\Gtests
python test_streaming.py
```

### 注意事项

- 每次测试前先拉取最新代码
- 测试结果反馈给开发者A
- 不需要创建分支，直接在main上测试
