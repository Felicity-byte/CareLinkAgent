## 六、发现的问题与解决方案

### 问题1: 控制台编码问题

**现象**: Windows PowerShell默认GBK编码，导致中文输出乱码

**解决方案**: 在测试脚本开头添加编码设置

```python
if sys.platform == 'win32':
    sys.stdout.reconfigure(encoding='utf-8', errors='replace')
    sys.stderr.reconfigure(encoding='utf-8', errors='replace')
```

### 问题2: 端口占用

**现象**: 服务启动时端口50053被占用

**解决方案**:

```powershell
Get-NetTCPConnection -LocalPort 50053 | Select-Object OwningProcess
Stop-Process -Id [PID] -Force
```

### 错误处理改进

**针对空响应问题的改进措施**:

在测试过程中发现，当AI服务返回空响应时，前端无法获知具体错误原因。为此，在 `service.py` 中添加了错误提示机制：

**改进内容**:

1. **空响应检测**: 在流式输出结束后检查 `full_response` 是否为空
2. **错误码返回**: 返回结构化错误信息，便于前端处理
3. **日志记录**: 在服务端记录警告日志

**错误返回格式**:

```json
{
    "content": "[ERROR:SERVICE_BUSY]",
    "is_final": true,
    "error": "SERVICE_BUSY",
    "error_message": "AI服务繁忙，请稍后重试"
}
```

**前端处理建议**:

```javascript
if (response.error) {
    switch (response.error) {
        case "SERVICE_BUSY":
            // 显示"服务繁忙，请稍后重试"
            break;
        case "API_ERROR":
            // 显示"服务异常，请联系管理员"
            break;
    }
}
```

**改进文件**: `GlmAI/zhipuGLM/service.py` - `normal_chat_stream` 函数

### 11.3 核心问题深度分析

#### 核心问题一：AI的"理解"与测试的"评估"不匹配

**问题本质**: 你的AI像一个健谈的普通人，而测试标准却要求它像一个严谨的医生。

**具体表现**:

- **识别率低 (0%-33%)**: AI会说"伤口周围有红肿和发热感"，但测试脚本只认识"伤口发炎"这个精确关键词。这导致AI的回答从语义上看是正确的，但在评估时被判为错误。

- **专业性评分低 (0%)**: AI会给出"这几天注意休息，吃点清淡的"这样的建议，它很口语化、很安全，但缺少"感染"、"换药"、"就医"等专业关键词，因此被判不合格。

#### 核心问题二：AI的"安全守则"与"医疗安全需求"冲突

**问题本质**: 通用AI被训练得"避免给出医疗建议"，而你的业务却需要它"果断给出安全警告"。

**具体表现**:

- **危急情况响应迟钝**: 当用户说"发烧39度+伤口流脓"时，通用AI的第一反应是"我不能提供医疗诊断，请咨询医生"，语气是建议性的。但医疗场景需要它第一句话就强硬地说："【紧急警告】请立即前往医院！"这种"强制警告"违背了通用AI的安全准则，所以如果不通过提示词强力扭转，它很难做到。

#### 核心问题三：数据流转存在"脆弱的断裂点"

**问题本质**: 从"对话"到"报告"的代码逻辑不够健壮，一旦格式有偏差，就会失败。

**具体表现**:

- **病历内容为空**: 在某一轮测试中，病历完全没生成。这大概率不是AI"忘了写"，而是你的后端代码在解析AI的回复时，因为格式、标点符号、换行等微小变化导致解析失败，又没有做好"失败回退"处理，最终数据库写入了空值。

- **响应时间波动大 (13秒 vs 55秒)**: 这通常指向网络问题、API限流或服务器资源竞争，而非AI模型本身的问题。代码中可能没有处理这些偶发性延迟的机制。
